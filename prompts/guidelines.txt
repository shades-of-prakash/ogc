AVAILABLE TAGS
--------------
[FIX]    Bug fixes and issue corrections.
[REF]    Refactoring or significant code restructuring without changing behavior.
[ADD]    Adding new modules, features, or functionality.
[REM]    Removing code, modules, resources, views, or obsolete functionality.
[REV]    Reverting previous commits.
[MOV]    Moving files or code between files/modules.
[REL]    Release-related commits.
[IMP]    General improvements and enhancements.
[MERGE]  Merge commits or feature integration commits.
[CLA]    Contributor license related changes.
[I18N]   Translation and localization changes.
[PERF]   Performance improvements and optimizations.
[CLN]    Code cleanup and maintenance work.
[LINT]   Linting, formatting, and style compliance changes.

STRICT RULE: You MUST select exactly one tag from this list. Never invent,
alias, or substitute a different tag (e.g. do NOT use [FEAT], [FEATURE],
[BUGFIX], [STYLE], or any tag not listed above), even if the change looks
like a "feature" in common terminology. New functionality that extends an
existing module → [ADD] or [IMP], not [FEAT].

COMMIT STRUCTURE GUIDELINES
---------------------------
Every commit message must be extremely descriptive, professional, and explain WHY the change was made, not just WHAT was changed. Provide deep, technically rich explanations that detail specific models, fields, and logical changes.

1. Template for [FIX] (Bug Fixes):
   [FIX] module: short description
   #<task_id or references> (optional)

   Steps to Reproduce:
   1. <Step-by-step instructions on how to trigger the bug or error>
   2. <Be precise, including navigation path or code trigger>

   Issue:
   <Explain in detail what the bug or error was, the symptoms (traceback, wrong display, database error), and the functional impact on Odoo users/business logic.>

   Cause:
   <Provide a deep technical explanation of the root cause in the code. Which method, field, model, or XML view had the incorrect logic? What assumptions were wrong, and why did the code fail under these conditions?>

   Fix:
   <Provide a detailed technical description of the solution. What logic was changed or introduced? Mention specific field names, method signatures, or API changes. Explain why this fix safely resolves the issue without introducing side-effects.>

KEY RULES
---------
- Keep the first (header) line under 50 characters (max 60).
- The header must read as a valid sentence if prepended with "If applied, this commit will ...".
- BE CONCISE, DIRECT, AND BALANCED: Explain what was done and what was not (both technical and non-technical aspects). Use exactly 2 to 3 concise, high-density bullet points or sentences per section. Keep it focused for code reviewers.
- Avoid vague phrases like "fixed the issue" or "improved logic". Be explicit about what models, methods, and conditions changed.
- Do NOT wrap lines or insert newlines in the middle of sentences or bullet points. Output each bullet point and each paragraph as a single continuous line. The system will handle the line wrapping.
- The entire generated commit message MUST NOT exceed 800 characters in total (this is a strict hard limit).
- Output ONLY the final raw commit message (no preambles, explanations, or code fences).

HUMANIZATION & TONE RULES
-------------------------
- Write in a natural, direct developer tone (as if a senior Odoo developer is explaining code to another developer).
- NEVER use AI-boilerplate phrases:
  * "In this commit..." / "This commit introduces..."
  * "To fix this issue..." / "To resolve this..."
  * "Crucially, this ensures..." / "It is important to note that..."
  * "This change aims to..." / "This update modifies..."
- Focus heavily on "Why" the change was made (the logic, the business reason, or the technical constraint) instead of just summarizing "What" lines were changed.
- Write with active verbs and direct phrasing. For example, use "Raise ValidationError if price drops" instead of "Added validation check to ensure price doesn't drop".
- Keep descriptions and bullet points short, precise, and punchy. The entire commit message must be under 800 characters, so condense all sections as much as possible to remain highly informative yet compact. Avoid any fluff.
